ARCPro: Passive Visual Protocol

Hierarchical Reinforcement Learning for Vision-Only Autonomous Intersection Navigation

Project Overview & The PVP

The goal is to get an F1TENTH car to navigate intersections using nothing but a front-facing camera.

The Passive Visual Protocol is how we enforce that. Signals like GPS, lane geometry, and path curvature are available in the simulator, but we deliberately zero them out in the observation vector. The policy only receives pixels and basic vehicle state. Lane detection still runs under the hood for computing rewards and detecting when the car leaves the road, but those values never make it into what the agent actually sees.

The reason this matters for sim-to-real is straightforward: if the agent never trains on signals that don’t exist on the real robot, there’s nothing to break when we deploy. The observation is the same in both places. We’re targeting the MiniCity setup in the lab, which has a couple 4-way and 3-way intersections with modular buildings.

What the agent gets each step:

  • A 160×90 RGB image (D435i camera, downsampled)
  • A 12-element telemetry vector: 8 slots are active (speed, yaw rate, last action, turn token, go/stop signal, cumulative distance) and 4 are zeroed out for PVP (goal distance, lateral offset, lane confidence, curvature)

Hierarchical Architecture

The core idea is splitting “where to go” from “how to drive.”

The Worker is a classical planner, no learning involved. It sits on top of an intersection graph and watches the car’s position. When the car approaches an intersection, the Worker checks what exits are available based on heading, picks a turn, and commits to it. It has three states: cruising, deciding, and committed. Once it picks a direction it holds that decision until the car clears the intersection. For training I can set it to follow a fixed route, sample randomly, or run a curriculum that biases toward going straight early on and adds more turns over time.

The Driver is the learned part. A CNN processes the camera image, fuses it with the telemetry vector, and feeds everything into an LSTM. From there the flow is sequential: the Planning Head takes the LSTM output and predicts small corrections on top of kinematic anchor curves that come from the Worker’s turn command blended with current steering. It’s initialized near zero so early in training the car just follows the anchors and gradually learns to adjust. Then the Control Head takes the LSTM output concatenated with the planned waypoints and outputs steering, throttle, and brake. The control decisions are conditioned on what the planner decided.

For deployment, a Topological EKF tracks position along graph edges in Frenet coordinates with drift resets at each intersection. This way the Worker doesn’t need GPS or global coordinates to know when the car reaches an intersection.

The training environment talks directly to the Omniverse Python API (60Hz physics, 30Hz rendering, 10Hz policy). ROS2 is only used at deployment.

Key Progress Since February

ROS2 decoupled from training. The environment now goes straight through PhysX and Replicator. No more middleware overhead or nondeterministic message timing.

Worker-Driver architecture. The old approach was injecting a random float for turn direction with no awareness of road structure. Now the Worker queries an intersection graph, picks a real turn, and injects a discrete token into the observation. There’s also a safety gate that overrides throttle if the Scheduler says to stop.

PVP enforcement. I found during a code review that lane detection values were being written directly to the observation vector. The policy could have learned to lean on lateral offset instead of actually learning from the camera. Fixed it by keeping lane detection internal to the environment and zeroing those observation slots.

Episode termination. The environment was hardcoding terminated=False, meaning episodes never ended early. The car could drive off the road and just sit there accumulating negative reward. I added stuck detection, off-road detection using internal lane confidence, and fall detection so the agent gets a clear signal when something goes wrong.

DAVE-2 baseline. I built out the full pipeline for the 2016 NVIDIA behavioral cloning CNN: data collection, dataset with augmentation, and training. It’s ready to run as the ablation comparison against our hierarchical approach.

Multi-simulator registry. A factory pattern lets training scripts switch between Isaac Sim and Gazebo through config with lazy imports. The Gazebo environment is written and import-tested with gz-transport bindings I built from source.
